## <I18n de="Modul 3 — Skills" en="Module 3 — Skills" />

<DemoBox step={5} title="Release-Notes-Skill anlegen" lang="bash" context="In einer Claude-Session ablegen, ohne Editor-Wechsel.">
{`mkdir -p .claude/skills/release-notes

cat > .claude/skills/release-notes/SKILL.md <<'MD'
---
name: release-notes
description: |
  Erstelle Release Notes aus Commits seit dem letzten Git-Tag.
  Use when the user says "release notes", "changelog", "what changed",
  oder auf Deutsch "release notes erstellen" / "changelog für die letzten X tage".
allowed-tools:
  - Bash(git log *)
  - Bash(git tag *)
---

# Release Notes

## Vorgehen

1. \`git tag --sort=-creatordate | head -1\` -> letzter Tag (oder leer)
2. Wenn Tag vorhanden: \`git log <tag>..HEAD --pretty=format:"%h %s"\`
   sonst: \`git log -50 --pretty=format:"%h %s"\`
3. Gruppiere Commits nach Conventional-Commit-Typ:
   - feat       -> "Features"
   - fix        -> "Fixes"
   - chore/docs -> "Misc"
   - andere     -> "Other"
4. Output als Markdown mit \`# Release <next-version>\`-Header.

## Output-Format

\`\`\`markdown
# Release <next-version>

## Features
- abc123 — feat: add foo

## Fixes
- def456 — fix: handle null email

## Misc
- ghi789 — docs: update README
\`\`\`
MD

git add .claude/skills/release-notes/
git commit -m "feat(skills): add release-notes"`}
</DemoBox>

## <I18n de="Modul 4 — Spec-Driven" en="Module 4 — Spec-Driven" />

<DemoBox step={11} title="Spec-Template anlegen" lang="bash" context="Realistisches Beispiel-Spec für Email-Login.">
{`mkdir -p docs/specs docs/plans

cat > docs/specs/email-login.md <<'MD'
# Spec: Email-Login

## Goal
User können sich mit Email + Password einloggen.

## Why
Existierender OAuth-Flow ist auf Mobile zu langsam und brüchig.
Die meisten Support-Tickets der letzten 6 Wochen drehen sich um Login-Probleme.

## Constraints
- Bcrypt für Hashing — keine eigene Crypto
- Bestandsuser müssen migrierbar sein (NULL password_hash = "force reset")
- Lockout-State persistiert in DB, nicht in-memory
- DSGVO: Login-Versuche nicht länger als 30 Tage logge

## Acceptance Criteria
- Test: erfolgreicher Login mit korrektem Password
- Test: 3 falsche Versuche -> Lockout für 60 Sekunden
- Test: nach Lockout-Zeit wieder einloggen funktioniert
- Test: forgot-password sendet Mail innerhalb 30 Sekunden
- Test: Migration füllt password_hash mit NULL ohne Datenverlust

## Anti-Goals
- Kein Magic-Link in dieser Iteration
- Kein 2FA jetzt (separater Sprint)
- Kein Social-Login-Refactoring

## Files involved (Schätzung)
- migrations/0042_password_hash.sql (neu)
- api/auth/signup.py
- api/auth/login.py
- api/auth/lockout.py (neu)
- api/users/model.py
- frontend/login.tsx
- tests/test_login.py (neu)
MD

git add docs/specs/ && git commit -m "docs: add email-login spec"`}
</DemoBox>

## <I18n de="Modul 5 — Sicherheit" en="Module 5 — Security" />

<DemoBox step={14} title="Pre-Commit-Hook gegen Test-Verfälschung" lang="bash" context="Verhindert dass Claude Tests verändert um sie grün zu kriegen.">
{`cat > .git/hooks/pre-commit <<'SH'
#!/bin/sh
# Block commits that touch test files without explicit "ALLOW_TEST_CHANGE"
# in the commit message.
test_changes=$(git diff --cached --name-only | grep -E '(^|/)tests?/|test_.*\\.py$|.*\\.test\\.(ts|tsx|js)$' || true)

if [ -n "$test_changes" ]; then
  msg=$(git log --format=%B -n 1 HEAD 2>/dev/null || echo "")
  staged_msg="$(cat \"$1\" 2>/dev/null || echo \"\")"
  if ! echo "$staged_msg$msg" | grep -q "ALLOW_TEST_CHANGE"; then
    echo "ERROR: test files modified without ALLOW_TEST_CHANGE in commit message:"
    echo "$test_changes"
    exit 1
  fi
fi
SH
chmod +x .git/hooks/pre-commit`}
</DemoBox>
