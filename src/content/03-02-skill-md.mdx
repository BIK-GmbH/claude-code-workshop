# <I18n de="Anatomie eines Skills" en="Anatomy of a Skill" />

<De>
## Verzeichnis-Struktur

```
.claude/skills/
└── pr-review/
    ├── SKILL.md           # Pflicht: Definition
    ├── checklist.md       # optional: Helper-Doku
    └── scripts/
        └── extract-diff.sh
```

## SKILL.md-Aufbau

<CommandBox lang="markdown">
{`---
name: pr-review
description: |
  Review a GitHub pull request systematically. Use when the user
  shares a PR URL or types "review PR #123".
allowed-tools:
  - Bash(gh pr view *)
  - Bash(gh pr diff *)
  - Read
  - Write
---

# PR Review Skill

## Step 1: Fetch the PR
\`\`\`bash
gh pr view $ARGUMENTS --json title,body,files,commits
\`\`\`

## Step 2: Read the diff
Apply the checklist in checklist.md (security, correctness, simplicity, tests).

## Step 3: Comment
Post structured review via \`gh pr review\`.`}
</CommandBox>

## Frontmatter-Felder

| Feld | Pflicht | Was es macht |
|---|---|---|
| `name` | ✓ | Skill-ID, auch Slash-Command-Name |
| `description` | ✓ | Trigger-Text — sollte typische User-Anfragen enthalten |
| `allowed-tools` | optional | Whitelist Tools, sonst geerbt |
| `model` | optional | erzwingt z.B. `opus-4-7` für diesen Skill |

<NoteCard variant="tip" title="Goldene Regel für Trigger">
Im `description` nicht nur sagen WAS der Skill tut, sondern auch HOW USERS ASK. Beispiele in Anführungszeichen, mehrere Sprachen wenn ihr DE+EN unterstützt.
</NoteCard>
</De>
