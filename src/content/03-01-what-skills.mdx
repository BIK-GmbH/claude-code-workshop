# <I18n de="Skills" en="Skills" />

<De>
## Definition

Ein **Skill** ist ein wiederverwendbares Stück „Wissen + Werkzeug" für Claude Code:

- Ein **Verzeichnis** mit einer `SKILL.md` + optionalen Helper-Files
- Frontmatter beschreibt **wann** der Skill greift (`description`) und wer ihn auslösen darf
- Body beschreibt **wie** Claude vorgehen soll
- Optional: Scripts, Templates, Beispiele

## Wo liegen Skills?

| Scope | Pfad | Wer hat Zugriff |
|---|---|---|
| Projekt | `.claude/skills/<name>/SKILL.md` | nur dieses Repo |
| User | `~/.claude/skills/<name>/SKILL.md` | alle Repos dieses Users |
| Plugin | über Plugin-Manager installiert | nach Install verfügbar |

## Wann greift ein Skill?

- **Auto-Invoke:** Frontmatter `description` matcht User-Anfrage → Claude lädt Skill automatisch
- **Manuell:** `/<skill-name>` als Slash Command
- **Über andere Skill:** ein Skill kann andere referenzieren
- **Manual only:** `disable-model-invocation: true` für Workflows mit Side Effects

## Beispiel — `daily-standup`

<CommandBox lang="markdown">
{`---
name: daily-standup
description: Generate a daily standup summary from recent git activity.
  Use when the user asks "what did I do yesterday" or "create my standup".
---

# Daily Standup

1. Run \`git log --author=$(git config user.email) --since=yesterday\`
2. Group commits by file area
3. Render as markdown:
   - **Yesterday:** <bullets>
   - **Today:** <to-do from open tasks>
   - **Blockers:** <none/list>`}
</CommandBox>

## Skill vs. Plugin — der Unterschied

Wichtige Begriffsklärung, weil oft verwechselt:

| | **Skill** | **Plugin** |
|---|---|---|
| Was es ist | **Inhalt** — eine `SKILL.md` mit Anweisungen / Workflow | **Container** — bündelt Skills, Subagents, MCP-Server, Hooks, Slash-Commands in EIN installierbares Paket |
| Granularität | Eine Aufgabe (`release-notes`, `code-review`) | Setup für einen ganzen Use-Case oder Tech-Stack |
| Distribution | Manuell ins `.claude/skills/`-Verzeichnis kopieren | `/plugin install <name>` aus einem **Marketplace** |
| Namespacing | Skill-Name global in der Session | Plugin-Skills sind namespaced: `/my-plugin:review` |
| Wann nutzen | Eine Routine die du selbst brauchst | **Setup-Reuse über mehrere Repos** oder **Distribution an andere** |

> **Ein Plugin enthält Null oder mehrere Skills.** Skill ist nicht eine Alternative zu Plugin — Skill ist ein Baustein **innerhalb** eines Plugins.

**Praxis-Trigger** (laut Anthropic):
- *Eine wiederkehrende Routine* → schreib einen **Skill**
- *Zweites Repo braucht das gleiche Setup* → pack es als **Plugin**
- *Mehrere Plugins teilen wollen* → Plugin in einem **Marketplace** hosten

Die meisten Cards in unserem **Top-Skills-Katalog** (Anhang 99.08) sind eigentlich **Plugins** — sie bringen mehrere Skills + oft MCP-Server + Hooks mit. Standalone-Skills (z.B. unser eigener `release-notes` oder `find-todo`) bleiben einzelne `SKILL.md`-Files.

<En>
## Definition

A **skill** is a reusable piece of "knowledge + tooling" for Claude Code:

- A **folder** with a `SKILL.md` + optional helper files
- Frontmatter describes **when** the skill triggers (`description`) and who may invoke it
- Body describes **how** Claude should proceed
- Optional: scripts, templates, examples

## Where do skills live?

| Scope | Path | Who has access |
|---|---|---|
| Project | `.claude/skills/<name>/SKILL.md` | this repo only |
| User | `~/.claude/skills/<name>/SKILL.md` | all repos for this user |
| Plugin | installed via plugin manager | available after install |

## When does a skill trigger?

- **Auto-invoke:** frontmatter `description` matches user query → Claude loads skill automatically
- **Manual:** `/<skill-name>` as slash command
- **From other skills:** a skill can reference others
- **Manual only:** `disable-model-invocation: true` for workflows with side effects

## Skill vs. Plugin — the difference

Often confused — clarify upfront:

| | **Skill** | **Plugin** |
|---|---|---|
| What it is | **Content** — one `SKILL.md` with instructions / workflow | **Container** — bundles skills, subagents, MCP servers, hooks, slash commands into ONE installable package |
| Granularity | One task (`release-notes`, `code-review`) | Setup for a whole use case or tech stack |
| Distribution | Manually copied into `.claude/skills/` | `/plugin install <name>` from a **marketplace** |
| Namespacing | Skill name is global in the session | Plugin skills are namespaced: `/my-plugin:review` |
| When to use | A routine you need yourself | **Setup reuse across repos** or **distribution to others** |

> **A plugin contains zero or more skills.** "Skill" is never an alternative to "plugin" — skill is a building block **inside** a plugin.

**Practical triggers** (per Anthropic):
- *A recurring routine* → write a **skill**
- *A second repo needs the same setup* → package it as a **plugin**
- *Want to share multiple plugins* → host the plugin in a **marketplace**

Most cards in our **Top Skills catalog** (appendix 99.08) are actually **plugins** — they ship multiple skills + often MCP servers + hooks. Standalone skills (e.g. our own `release-notes` or `find-todo`) stay as single `SKILL.md` files.
